# 📚 学习笔记：LeetCode 41. 缺失的第一个正数

##  题目描述

给你一个未排序的整数数组 `nums`，请你找出其中 **没有出现的最小正整数**。

**要求：**

- 时间复杂度为 $O(n)$
- 只使用常数级别额外空间

------

## 示例

### 示例 1

> **输入：** `nums = [1,2,0]`
>
> **输出：** `3`
>
> **解释：** `1` 和 `2` 都在数组中，最小缺失正数是 `3`。

### 示例 2

> **输入：** `nums = [3,4,-1,1]`
>
> **输出：** `2`
>
> **解释：** `1` 在数组中，但 `2` 不在数组中。

### 示例 3

> **输入：** `nums = [7,8,9,11,12]`
>
> **输出：** `1`
>
> **解释：** 最小正数 `1` 没有出现。

------

## 题目本质

这题看起来是在“找一个数”，但本质上是在问：对于长度为 `n` 的数组，最小缺失正数可能落在哪个范围？

**结论：只可能在 $1 \sim n+1$ 之间。**

- 如果数组中没有 `1`，答案就是 `1`。
- 如果数组中有 `1, 2, 3, ..., n`，那么答案就是 `n+1`。
- 因此，我们不需要关心：
  - 小于等于 `0` 的数。
  - 大于 `n` 的数。

------

##  核心思路：原地哈希 / 置换到位

最优解法是：**把值为 $x$ 的数尽量放到下标 $x-1$ 的位置上。**

也就是说：

- `1` 应该放在 `nums[0]`
- `2` 应该放在 `nums[1]`
- `3` 应该放在 `nums[2]`
- ...
- `n` 应该放在 `nums[n-1]`

这样做完以后，再扫描一遍数组：

- 第一个不满足 `nums[i] == i + 1` 的位置，答案就是 `i + 1`。
- 如果全部匹配，答案就是 `n + 1`。

------

##  为什么能这样做？

对于一个长度为 `n` 的数组，我们只关心 $1 \sim n$。每个合法数字 $x$ 都有一个“应该待的位置”：$x \rightarrow x - 1$。

我们可以把数组本身当成一个“哈希表”：

- 数值是 Key。
- 下标位置是它应该去的槽位。

这就是所谓的：**原地哈希 (In-place Hash)** 或 **Cyclic Sort 思想**。

------

##  标准算法流程

### 第一步：遍历数组，把合法数字放到正确位置

对于当前元素 `nums[i]`，如果它满足：

- $1 \le nums[i] \le numsSize$
- 它应该去的位置是 `nums[i] - 1`

不断交换，直到：

1. 当前值不在 $1 \sim numsSize$ 范围内。
2. 当前值已经在正确位置。
3. **目标位置已经有相同值（避免死循环）**。

### 第二步：扫描数组找答案

- 如果 `nums[i] != i + 1`，说明 `i + 1` 缺失，直接返回。
- 如果所有位置都满足，返回 `numsSize + 1`。

------

##  C 语言代码实现

C

```
int firstMissingPositive(int* nums, int numsSize) {
    for (int i = 0; i < numsSize; i++) {
        // 使用 while 循环不断交换，直到 nums[i] 到达正确位置或不合法
        while (nums[i] >= 1 && 
               nums[i] <= numsSize && 
               nums[nums[i] - 1] != nums[i]) {
            // 交换 nums[i] 到它应该去的位置 nums[i] - 1
            int target = nums[i] - 1;
            int temp = nums[target];
            nums[target] = nums[i];
            nums[i] = temp;
        }
    }

    // 二次遍历寻找缺失值
    for (int i = 0; i < numsSize; i++) {
        if (nums[i] != i + 1) {
            return i + 1;
        }
    }

    return numsSize + 1;
}
```

------

##  执行过程示例

### 示例一：`nums = [1,2,0]`

- `i = 0`: `nums[0]=1`，已在位置 0，不动。
- `i = 1`: `nums[1]=2`，已在位置 1，不动。
- `i = 2`: `nums[2]=0`，不合法，忽略。
- **扫描：** `nums[2] != 3`，返回 **3**。

### 示例二：`nums = [3,4,-1,1]`

- `i = 0`: `3` 换到位置 2 $\rightarrow$ `[-1,4,3,1]`，`-1` 不合法，停止。
- `i = 1`: `4` 换到位置 3 $\rightarrow$ `[-1,1,3,4]`；继续处理 `nums[1]=1`，换到位置 0 $\rightarrow$ `[1,-1,3,4]`。
- **扫描：** `nums[1] != 2`，返回 **2**。

------

##  为什么你原代码会超时？

原因在于：**没有判断目标位置上是否已经有相同的值。**

如果数组为 `[1,1]`，当处理 `i = 1` 时：

- `nums[1] = 1` 想去位置 0。
- 但位置 0 已经是 `1`。
- 交换后数组不变，导致 `while` 循环无限执行，产生超时（TLE）。

------

## 关键防死循环条件

必须加上：`nums[nums[i] - 1] != nums[i]`。

> 只有目标位置上的值不是它自己时，才进行交换。

------

##  时间复杂度为什么是 $O(n)$？

虽然嵌套了 `while`，但：**每次交换至少会把一个数字放到它最终应该在的位置。**

每个元素最多被交换归位一次，总交换次数是线性的，所以总时间复杂度为 $O(n)$。

------

##  复杂度分析

- **时间复杂度：** $O(n)$
- **空间复杂度：** $O(1)$

------

## 易错点总结

- **13.1 范围判断：** 只处理 $1 \le nums[i] \le numsSize$。
- **13.2 重复元素：** 必须在交换前判断目标位是否已有相同值。
- **13.3 排序局限：** 排序复杂度为 $O(n \log n)$，不符合 $O(n)$ 要求。
- **13.4 空间局限：** 额外开辟哈希表会导致空间复杂度变为 $O(n)$。

------

##  这道题的模板思想

看到“数组长度为 $n$”、“数值与下标有对应关系”、“要求 $O(n)$ 时间 $O(1)$ 空间”时，优先考虑：**把元素放到它应该去的位置。**

------

##  面试答题模板

1. 说明最小缺失正数一定在 $1 \sim n+1$ 范围内。
2. 提出“原地哈希”思路：将 $1 \sim n$ 的数字 $x$ 交换到下标 $x-1$。
3. 强调 $O(1)$ 空间特性和处理重复元素的死循环判断。
4. 最后遍历数组找第一个不匹配的索引。

------

##  一句话总结

> **将 $1 \sim n$ 范围内的数字原地放到对应下标位置，最后找第一个 `nums[i] != i + 1` 的位置。**
