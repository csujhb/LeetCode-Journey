# 学习笔记：下一个排列 (LeetCode 31)

---

## 核心问题解析

寻找“下一个排列”本质上是寻找字典序刚好比当前排列大一点点的排列。为了保证增加的幅度尽可能小，我们需要在尽可能靠右的位置发生变动，将一个较小的数替换为一个稍大的数，并且将变动位置右侧的数字重新整理为字典序最小的升序状态。

---

## 常见方法一：两遍扫描法（绝对最优解）

这是解决本题的标准算法，完美满足了时间复杂度和空间复杂度的严苛要求。

**思路剖析：**
整个过程可以通过几个清晰的阶段来完成，利用从后向前的贪心策略，完全不需要额外的存储空间。

* **定位较小数：** 从数组的末尾开始向前遍历，寻找首个满足前一个数字严格小于后一个数字的相邻元素对。记录这个前一个数字的索引。如果一直遍历到数组开头都没有找到，说明整个数组是单调递减的，即当前已经是最大字典序排列。
* **定位稍大数并交换：** 如果成功找到了上述的较小数，说明可以生成更大的排列。此时再次从数组末尾开始向前遍历，在较小数的右侧找到首个大于该较小数的元素。找到后，将这两个数字互换位置。这一步操作确保了较高的权重位换到了一个稍大的数字，整体字典序随之变大。
* **重置后缀序列：** 交换完成后，较小数原本位置右侧的序列依然保持着严格递减的性质。为了让字典序增大的幅度最小（即紧挨着的“下一个”），必须将这一段降序子数组反转，使其变为单调递增的升序排列。对于原本就是最大排列的情况，整体反转也能恰好得到字典序最小的起始排列。

**C语言实现代码：**
```c
// 辅助函数：交换指针指向的两个元素
void swap(int *a, int *b) {
    int temp = *a;
    *a = *b;
    *b = temp;
}

// 辅助函数：反转数组中的指定区间
void reverse(int *arr, int left, int right) {
    while (left < right) {
        swap(&arr[left], &arr[right]);
        left++;
        right--;
    }
}

void nextPermutation(int* nums, int numsSize) {
    int i = numsSize - 2;
    
    // 从后向前寻找第一个相邻升序对，定位较小数
    while (i >= 0 && nums[i] >= nums[i + 1]) {
        i--;
    }
    
    // 如果找到了较小数，再次从后向前寻找稍大数并交换
    if (i >= 0) {
        int j = numsSize - 1;
        while (j >= 0 && nums[i] >= nums[j]) {
            j--;
        }
        swap(&nums[i], &nums[j]);
    }
    
    // 将较小数右侧的递减序列反转为递增序列
    // 如果原数组是完全递减的，i 会等于 -1，此时正好反转整个数组
    reverse(nums, i + 1, numsSize - 1);
}
```

## 常见方法二：暴力穷举法（概念解）

在没有接触过规律推导时，这是最符合直觉的思考方式，但通常不作为实际编码方案。

**思路剖析：**
* 生成给定数组元素的所有可能的全排列组合。
* 将产生的所有排列组合按照字典顺序从小到大进行全面排序。
* 在排好序的集合中查找当前给定的输入排列，查找到之后，直接提取位于它后一位的排列作为答案。如果是序列中的最后一个，则提取首个排列。
* 由于全排列的数量是阶乘级别的，这种思路的时间和空间开销极大，一旦数组长度稍微增加就会导致超时和内存溢出。因此这种解法仅停留于理论认知层面，无法用于通过在线测试。

---

## 深度总结

这道题的核心难点在于突破表象，发现字典序递增背后的数学规律。**两遍扫描法**巧妙地利用了数组的升降序特性，在最坏情况下也仅仅只需遍历数组两次，并执行一次局部的双指针反转。

这种方法的精妙之处在于它不仅将时间复杂度稳稳控制在 O(n)，而且全程都在原数组内存块上进行指针操作，空间复杂度严格保持在 O(1)。掌握这种从后向前的局部贪心与后缀处理思想，不仅能轻松攻克本题，对于未来解决其他涉及排列组合与字典序生成的算法题，也会有非常深远的启发。
