# 算法学习笔记：寻找旋转排序数组的最值 (LeetCode 153 / 154 深度解析)

## 🏷️ 核心标签

`#二分查找 (Binary Search)` `#布尔数组映射 (Boolean Mapping)` `#寻找左右边界` `#静态与动态锚点`

## 📖 原题描述

已知一个长度为 `n` 的数组，预先按照升序排列，经由 `1` 到 `n` 次 **旋转** 后，得到输入数组。例如，原数组 `nums = [0,1,2,4,5,6,7]` 在变化后可能得到：

* 若旋转 4 次，则可以得到 `[4,5,6,7,0,1,2]`
* 若旋转 7 次，则可以得到 `[0,1,2,4,5,6,7]`

给你一个元素值 **互不相同** 的数组 `nums` ，它原来是一个升序排列的数组，并按上述情形进行了多次旋转。请你找出并返回数组中的 **最小元素** （或延伸提问：**最大元素**）。
你必须设计一个时间复杂度为 $O(\log n)$ 的算法解决此问题。

## 🎯 核心破题思想：布尔映射 (Boolean Mapping)

解这类题的最高阶思维不是直接比较数字大小，而是“二分条件”。
我们可以找出一个判断条件，将杂乱的旋转数组映射为一个完美的 `[False, False, ..., True, True]` 布尔数组。

* **最小值** = 布尔数组中**第一个 `True**` （寻找左边界）。
* **最大值** = 布尔数组中**最后一个 `True**` （寻找右边界）。

---

## 💻 场景一：寻找最小值 (左边界模型)

**映射条件：** 该元素是否小于等于最右侧元素？（即 `x <= nums[right]` 或 `x <= nums[n-1]`）
**举例推演：** `[4, 5, 6, 7, 0, 1, 2]` 映射为 `[F, F, F, F, T, T, T]`，我们要找第一个 `T`。

```c
int findMin(int* nums, int numsSize) {
    int left = 0;
    int right = numsSize - 1;

    while (left < right) {
        int mid = left + (right - left) / 2;
        
        // 映射条件为 False (说明还在左边的大数山上)
        if (nums[mid] > nums[right]) {
            left = mid + 1; // 跨过 mid，去右边找第一个 True
        } 
        // 映射条件为 True (说明已经到了右边的小数山上)
        else {
            right = mid;    // mid 可能是目标，绝不能跳过
        }
    }
    return nums[left];
}

```

## 💻 场景二：寻找最大值 (右边界模型)

### 策略 A：工程复用法 (极力推荐)

最大值永远紧挨在最小值的左侧。找到最小值的索引后，向左偏移一位即可（注意处理越界）。

```c
// 假设已经用上面的逻辑找到了最小值的索引 left
int maxIndex = (left - 1 + numsSize) % numsSize;
return nums[maxIndex];

```

### 策略 B：直球对决法 (真正的右边界二分)

**映射条件：** 该元素是否大于等于最左侧元素？（即 `x >= nums[0]`）
**举例推演：** `[4, 5, 6, 7, 0, 1, 2]` 映射为 `[T, T, T, T, F, F, F]`，我们要找最后一个 `T`。

```c
int findMax(int* nums, int numsSize) {
    int left = 0, right = numsSize - 1;
    if (nums[left] < nums[right]) return nums[right]; // 处理完全升序的特例

    while (left < right) {
        // ⚠️ 致命陷阱：寻找右边界时，mid 必须向上取整 (+1)，否则死循环！
        int mid = left + (right - left + 1) / 2; 
        
        // 映射条件为 True
        if (nums[mid] >= nums[0]) {
            left = mid;      // mid 可能是最后一个 True，绝不能跳过
        } 
        // 映射条件为 False
        else {
            right = mid - 1; // 越过 False，向左收缩
        }
    }
    return nums[left];
}

```

## 🧠 灵魂发问：参照物（锚点）到底该选谁？

在编写判断条件时，我们经常纠结是用 `nums[left]`、`nums[right]`、`nums[0]` 还是 `nums[n-1]`。这里有极深的算法学问：

### 绝对静态锚点 (`nums[0]` 或 `nums[n-1]`)

* **优点**：逻辑最严密。它们是划分“左山”和“右山”的绝对基准线，比较条件（尺子的刻度）在整个循环中永远不变。
* **适用场景**：寻找最大值时，强烈建议使用 `nums[0]`；无重复元素的数组中，使用 `nums[n-1]` 找最小值完美无缺。

### 安全的动态锚点 (`nums[right]`)

* **现象**：全网寻找最小值的题解，几乎清一色使用的是 `nums[mid] > nums[right]`。
* **原因**：
1. **局部等价**：只要最小值还在区间内，`right` 指针就永远停留在“右侧的小数山”上，用它作为参照物是安全的。
2. **向下兼容（终极原因）**：为了兼容 **LeetCode 154 (允许数组有重复元素)**。当遇到 `nums[mid] == nums[right]` 这种无法分辨左右山的极端情况时，动态锚点允许我们执行 `right--` 来保守缩小区间，而静态锚点遇到这种情况会彻底卡死。



### 危险的动态锚点 (`nums[left]`)

* **原因**：在找最大值时，如果使用 `nums[mid] >= nums[left]`，由于 `left` 会不断向右攀爬爬上“大数山”，这个参照基准会不断变大，导致布尔映射关系被破坏。这会极大增加边界推理的认知负担，极易写出死循环（Bug）。

## 🧭 进阶：四大场景下的锚点选择指南

在编写二分查找的判断条件时，我们经常面临使用 `nums[left]`、`nums[right]`、`nums[0]` 还是 `nums[n-1]` 作为比较基准（锚点）的抉择。

记住核心法则：**找最小值盯住右侧（`right` 或 `n-1`），找最大值盯住左侧（`left` 或 `0`）。**

### 场景 1：寻找最小值（无重复值 - LeetCode 153）

**目标：寻找右半边小数阵营的“左边界”。**

* ✅ **`nums[right]`（最推荐/通用模板）**：动态安全锚点。无论怎么收缩，`right` 永远在右侧的小数山上，判定基准稳定，且能天然兼容完全升序的数组。
* ✅ **`nums[n-1]`（完全可行）**：静态绝对锚点。右侧阵营的绝对代表，逻辑严密（需前置拦截完全升序的特例）。
* ❌ **`nums[left]`（逻辑破坏）**：`left` 指针会不断向高处攀爬，导致判定基准变大，破坏二分的单调性。

### 场景 2：寻找最小值（有重复值 - LeetCode 154）

**目标：处理 `[3, 3, 1, 3]` 这种首尾相等导致二分“失明”的极端情况。**

* ✅ **`nums[right]`（唯一解法）**：当 `nums[mid] == nums[right]` 时，我们无法判定方向，但可以安全地执行 `right--`（扔掉边缘的重复项）来打破僵局，因为 `mid` 是该值的完美替身。
* ❌ **`nums[n-1] / nums[0]`（彻底失效）**：静态锚点是固定的死值，无法执行收缩操作，遇到上述情况必定陷入死循环。
* ❌ **`nums[left]`（逻辑破坏）**：同场景 1。

### 场景 3：寻找最大值（无重复值）

**目标：寻找左半边大数阵营的“右边界”。**

* ✅ **`nums[0]`（最推荐）**：静态绝对锚点。左侧阵营的绝对代表。只要 `mid >= nums[0]`，说明一定还在左侧的大数山上，逻辑最清晰，认知负担最小。
* ⚠️ **`nums[left]`（可行但不推荐）**：虽然也能跑通，但由于 `left` 是个动态攀升的基准，在脑海中推演边界时极易把自己绕晕。
* ❌ **`nums[right]`（逻辑破坏）**：`right` 会不断向低处滑落，基准变小，破坏二分逻辑。

### 场景 4：寻找最大值（有重复值）

**目标：应对重复元素导致的判定失效（与场景 2 完全对称）。**

* ✅ **`nums[left]`（唯一解法）**：当遇到 `nums[mid] == nums[left]` 无法判定时，可以安全地执行 `left++` 来保守缩小区间（寻找右边界时 `mid` 靠右取整，跳过 `left` 不会丢失替身）。
* ❌ **`nums[0] / nums[n-1]`（彻底失效）**：固定锚点无法执行 `left++` 操作，卡死。
* ❌ **`nums[right]`（逻辑破坏）**：同场景 3。

---

## 💡 终极实战口诀 (Golden Rule)

> **“无重动静皆可，有重必用动态。”**

* **没有重复值时：** 你想怎么秀理解都可以，静态标尺、动态标尺都能完美解题。
* **有重复值时：** 必须放弃静态锚点（`0` 和 `n-1`）。找最小值用 `nums[right]` 配合 `right--`；找最大值用 `nums[left]` 配合 `left++`。
