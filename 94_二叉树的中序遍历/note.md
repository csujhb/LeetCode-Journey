# LeetCode 94. 二叉树的中序遍历 - 学习笔记

## 核心概念理解

**中序遍历 (Inorder Traversal)** 的访问顺序是：**左子树 -> 根节点 -> 右子树**。
对于二叉搜索树（BST）来说，中序遍历的输出结果恰好是一个**递增的有序序列**。

在 C 语言解题中，这道题的一个核心难点在于：**如何在递归或循环中维护一个全局的数组索引？**
> **解决方案**：传递指向索引变量的指针（如 `int* returnSize`）。每次往数组存入数据后，通过 `(*returnSize)++` 来更新其实际值，确保所有底层函数调用共享并修改同一个内存地址里的游标。

---

## 三种经典解法拆解

### 递归法 (Recursive)
最自然、最符合树结构的解法，也是面试中最基础的要求。
* **核心思路**：将大树的遍历拆解为左、中、右三个基本步骤的自我调用。
* **时间复杂度**：$O(N)$，每个节点都会被访问恰好一次。
* **空间复杂度**：$O(H)$（$H$ 为树的高度）。取决于隐式系统调用栈的深度。
* **核心代码**：
    ```c
    void inorder(struct TreeNode* root, int* res, int* returnSize) {
        if (!root) return;
        inorder(root->left, res, returnSize);
        res[(*returnSize)++] = root->val; // 核心：存入数据并自增索引
        inorder(root->right, res, returnSize);
    }
    ```

### 迭代法 (Iterative / 显式栈)
用自己开辟的数组模拟系统的函数调用栈。
* **核心思路**：
    * 只要当前节点不为空，就一直往左走，并将沿途节点压入栈中。
    * 当走到尽头（左子树为空）时，从栈顶弹出一个节点，访问它（存入结果数组）。
    * 将游标转向该节点的右子树，继续上述的循环过程。
* **时间复杂度**：$O(N)$。
* **空间复杂度**：$O(H)$，显式栈的最大深度为树高。
* **适用场景**：系统严格限制递归深度（防止 Stack Overflow）时，或者面试官明确要求不准使用递归时。

### Morris 遍历 (空间极客解法)
打破常规，用时间（极少量的额外遍历）换取极致的空间，是这道题的“加分项”。
* **核心思路**：利用树底层叶子节点大量闲置的 `right` 指针，将其临时指向中序遍历的后继节点（即当前节点的祖先），形成“线索”。遍历结束后再拆除线索，恢复原树的原貌。
* **找前驱节点的规则**：当前节点左子树中，最右下角的那个节点。
* **时间复杂度**：$O(N)$。虽然寻找前驱的过程看起来有嵌套 `while` 循环，但整棵树的每条边最多只会被走两次（建线索一次，拆线索一次），均摊下来依然是严格的线性时间。
* **空间复杂度**：$O(1)$。除了结果数组，只使用了两个游标指针（`curr` 和 `pre`），完全抛弃了栈！
* **适用场景**：对内存空间要求极其苛刻的底层开发或嵌入式系统。

## 综合对比总结

| 解法 | 时间复杂度 | 空间复杂度 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- | :--- |
| **递归法** | $O(N)$ | $O(H)$ | 代码极其简洁，可读性最高 | 在极端倾斜的树中可能导致栈溢出 |
| **迭代法** | $O(N)$ | $O(H)$ | 避免了系统栈溢出的风险 | 代码相对繁琐，依然需要消耗额外空间 |
| **Morris遍历** | $O(N)$ | $O(1)$ | 空间利用率达到极致 | 逻辑极度复杂，运行过程中会暂时改变树的结构 |

## 个人复习备忘录
* **C 语言数组赋值神仙写法**：`res[(*returnSize)++] = val;` 务必熟练掌握。它等价于先 `res[*returnSize] = val;` 然后再 `(*returnSize) = (*returnSize) + 1;`。
* 遇到树的遍历，第一反应写递归保底；如果面试官要求优化空间，直接抛出 Morris 遍历降维打击。
* 写迭代法时，`while (curr != NULL || top != -1)` 的双重判断条件是精髓，千万不能漏掉栈不为空的判断。
